# Lean Gherkin 設計レビュー

この文書は `design.md` に対する批判的検討、リスク、未決定事項をまとめる。

## 1. 批判的な指摘

### 1.1 設計文書の責務は整理されたが、境界の継続管理が必要である

元の `design.md` は、設計、ロードマップ、実装順序、リスク、将来構想、リポジトリ構成案が 1 ファイルに混在していた。そのため、現在何を実装すべきか、何が将来検討なのか、何が設計上の制約なのか、どこまでがリポジトリ運用上の約束なのかが読み取りにくかった。

現在は文書分割によって責務が改善されている。一方で、今後の追記で再び `design.md` に実装計画や運用方針が混入しないように、文書間の境界を維持する必要がある。

対応:

- 設計本体は `design.md` に残す
- 仕様と想定記法は `spec.md` に分離する
- 実装計画は `implementation-plan.md` に分離する
- 将来拡張は `roadmap.md` に分離する
- 批判的検討とリスクはこの文書に分離する
- リポジトリ構成案は `repository-structure.md` に分離する

追加指摘:

- `design.md` の「段階的実装」は設計方針として残せるが、具体的な作業順序やマイルストーンは `implementation-plan.md` に寄せるべきである
- DSL の仕様、初期記法、採用しない記法の話は `spec.md` に寄せるべきである
- ファイル配置、モジュール分割、テスト配置の話は `repository-structure.md` に寄せるべきである
- 将来機能の列挙は、設計上の前提でない限り `roadmap.md` に寄せるべきである

### 1.2 Lean 構文として成立するかの検証が不足している

`syntax "scenario " str " do " gherkinStep* : gherkinScenario` のような案は方向性としては分かるが、実際の Lean parser で期待通りにネストや改行を扱えるかは未検証である。

指摘:

- Gherkin らしい見た目と Lean の構文制約の優先順位を明確にする必要がある
- `do` ブロック内の独自カテゴリがどの程度自然に書けるかを早期にプロトタイプで確認すべきである
- 構文案は「設計確定」ではなく「実験対象」として扱うべきである

### 1.3 `And` / `But` の意味解決が曖昧である

`StepKind` に `and` / `but` を独立した kind として持たせる案は単純だが、Gherkin の意味論では直前の `Given` / `When` / `Then` を継承する性質がある。

指摘:

- AST に入力上の kind と正規化後の kind の両方を持たせるかを決める必要がある
- step definition 解決時に `and` / `but` をどう扱うかが未定義である
- エラーメッセージでは元のキーワードを保持した方がよい

### 1.4 Step Definition の型安全性が最も難しい部分として過小評価されている

`step_def "I add {x:Int} and {y:Int}" => ...` は魅力的だが、文字列パターン、型解決、文字列から Lean 値への変換、handler の型検査を同時に要求する。

指摘:

- 初期実装では完全一致のみとし、型付き placeholder は明確に後続フェーズへ送るべきである
- placeholder の型は `FromString` 的な仕組みでよいのか、独自 parser typeclass が必要かを決める必要がある
- elaboration 時にどこまで検証し、実行時にどこから失敗させるかを分ける必要がある

### 1.5 実行モデルと Lean のビルドモデルの境界がまだ弱い

`#run_feature` / `#run_scenario` による明示実行方針は妥当だが、Lean の elaboration 中に IO を走らせること、Lake ビルド時の再現性、import 時副作用の扱いは設計の中心課題である。

指摘:

- `feature` 登録は純粋なメタデータ登録に限定する方針を明文化すべきである
- runner は通常の import では実行されないことを保証する必要がある
- テストランナーとして使う場合、Lake との統合単位を別途設計する必要がある

### 1.6 証明統合は価値が高いが抽象度がまだ高すぎる

`Given` を仮定、`When` を変換、`Then` を結論に対応させる構想は Lean らしいが、一般の Gherkin scenario を自然に `Prop` 化できるとは限らない。

指摘:

- テスト実行 DSL と証明 DSL は同じ syntax に載せる前に責務を分けて検討すべきである
- `Then` の接続先を命題、決定可能命題、テスト assertion、IO action のどれにするか整理が必要である
- 証明統合は最小実装の成功条件に含めない方がよい

### 1.7 エラー診断の設計が実装後回しになりやすい

自然言語風 DSL では、未定義 step、重複 step definition、構文位置の表示が品質を大きく左右する。

指摘:

- AST の初期段階から source position を持つか、後付けするかを決める必要がある
- 「似ている step definition 候補」を出すには文字列距離などの方針が必要である
- 重複定義では両方の定義位置を表示する必要がある

### 1.8 現在の実装実験と設計文書の同期が必要である

`LeanGherkin.lean` 側では `Feature:` 風の構文実験が進んでいるが、`design.md` の初期構文は文字列ベースの `feature "..." do` を優先する方針になっている。実験自体は有用だが、設計上の既定方針とプロトタイプの試行錯誤が混ざると、どの構文を正式採用するのかが曖昧になる。

指摘:

- `Feature:` 風構文は、正式設計ではなく構文プロトタイプとして位置付けるべきである
- 採用候補の構文ごとに、Lean parser で扱える範囲、エラー表示、拡張性を比較する必要がある
- 実装が先行した構文を追認するのではなく、設計上の判断基準を明文化してから採否を決めるべきである

## 2. 優先して避けるべき複雑性

初期実装では以下を避ける。

- 自然言語そのものの高度な解析
- 本家 Gherkin の完全互換
- 複雑な indentation-sensitive parser
- 多言語キーワード対応
- Examples / Scenario Outline の完全実装
- 非同期実行
- 外部プロセスとの統合

まずは Lean 上で安定して扱える DSL を作ることを優先する。

## 3. リスクと検討事項

### 3.1 Lean 構文との衝突

`feature`, `scenario`, `given`, `when`, `then` などのキーワード風構文が既存構文や将来の Lean 構文と衝突する可能性がある。

対策:

- namespace を設ける
- `gherkin` ブロック内限定にする
- prefix を導入する

例:

```lean4
gherkin_feature "Calculator" do ...
```

### 3.2 Indentation と構文パース

Gherkin は本来 indentation に依存した読みやすい構文だが、Lean の parser と完全に同じ感覚では扱えない可能性がある。

対策:

- 初期実装では `do` ブロックと文字列を使う
- 本家 Gherkin 風構文は後回しにする

### 3.3 実行とコンパイルの境界

Lean の elaboration 時に IO 実行を行う設計は慎重に扱う必要がある。

対策:

- 仕様登録と実行を分離する
- `#run_feature` のような明示的コマンドで実行する
- 通常の import 時に副作用を起こさない

### 3.4 エラーメッセージ品質

DSL ではユーザーが自然言語的に記述するため、エラー箇所が分かりにくくなりやすい。

対策:

- source position を保持する
- 未定義 step の候補を表示する
- 重複定義の位置を表示する
- feature / scenario 名を含めて報告する

## 4. 未決定事項

- `feature` などの短い構文をトップレベルに出すか、`gherkin` prefix / block でスコープを限定するか
- `Feature:` 風構文を正式候補として扱うか、プロトタイプに留めるか
- `And` / `But` を AST 上で正規化するか、元の kind として保持するか
- Step Definition の handler 型を `IO Unit`、独自 monad、命題、またはそれらの組み合わせのどれにするか
- `Feature` / `Scenario` の重複名を許可するか
- source position を最小 AST に含めるか、診断用 metadata として別管理するか
- 文書間の責務境界をどの程度厳密に運用するか
